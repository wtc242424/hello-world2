# Outline why developers use version control (e.g., tracking changes, collaboration, rollback, accountability).  (AC1.1) 

Version control is a system that helps developers manage and track changes to their code over time. It is an essential tool in software development because it allows teams to work efficiently and safely on the same project.

One reason developers use version control is tracking changes. Every modification to the code can be recorded, creating a history of the project. This makes it easy to see what changes were made and when they occurred.

Version control also supports collaboration. Multiple developers can work on different parts of the same project at the same time without overwriting each other's work. Changes can later be combined into a shared version of the project.

Another important benefit is rollback. If a bug or mistake is introduced, developers can return to a previous version of the code that was working correctly. This reduces the risk of losing progress and helps resolve issues quickly.

Finally, version control provides accountability. Every change is linked to a specific developer and commit message, making it clear who made a change and why. This helps teams review work, identify issues, and maintain responsibility for project contributions.


# Explain ways to use version control in a small Python project (e.g., commit often, meaningful messages, branches for features, pull requests for review). (AC1.2) 

In a small Python project, version control can be used in several ways to improve organisation and code quality.

A good practice is to commit changes often. Making small, regular commits creates a clear history of development and makes it easier to identify problems if they occur.

Developers should write meaningful commit messages that describe the purpose of each change. For example, "Add input validation to survey script" is much more useful than "Update code". Clear messages help other team members understand the project's history.

It is also useful to create branches for new features or bug fixes. For example, a branch called feature-report-generator could be used to develop a reporting function without affecting the main version of the project. Once testing is complete, the branch can be merged back into the main branch.

Another recommended practice is using pull requests. Before changes are merged, team members can review the code, suggest improvements, and check for errors. This helps maintain code quality and encourages collaboration.

By committing regularly, using descriptive commit messages, working on separate branches, and using pull requests for review, developers can manage a Python project more effectively and reduce the risk of errors.
